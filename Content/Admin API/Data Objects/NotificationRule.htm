NotificationRule

[Admin API Object]

A list of NotificationRule objects is returned as the response payload for this Admin API operation:

  • GET Notification Rules

The NotificationRule object is required as the request payload for these Admin API operations:

  • POST Updated Notification Rule
  • PUT New Notification Rule

The NotificationRule object consists of the attributes below. Each attribute description indicates whether the attribute is mandatory or optional when this object is transmitted as a request payload.

condition

(Mandatory, string) Comparator used in defining this rule: can be ">", "<", or "=".

 conditionVal

(Mandatory, string) Value against which to compare. The value of the statistic specified by statId will be compared to the conditionVal to determine whether a notification is called for. This statistic monitoring occurs on each node.

For example for a rule that triggers notifications if CPU utilization on any individual node exceeds 90%, the "statId" would be "cpu", the "condition" would be ">", and the "conditionVal" would be ".9".

For information about the conditionVal format appropriate for each statId see the table below.

 email

(Optional, string) Comma-separated list of email addresses to receive notifications. Or to use the default email address list (as configured on the CMC page Cluster → Cluster Config → Configuration Settings), set this to the string "default".

To not have email notifications as part of the rule (for instance, if the rule is only meant to trigger SNMP traps), set the "email" attribute to empty ("").

This defaults to empty ("") if the attribute is omitted in a PUT New Notification Rule request.

 enabled

(Optional, boolean) Whether the rule is enabled, true or false.

To disable an existing rule use POST Updated Notification Rule and set this attribute to false. (Or, alternatively, you can enable/disable a rule by using the POST Notification Rule Enable/Disable method.)

This attribute defaults to false if the attribute is ommitted in a PUT New Notification Rule request body.

 region

(Optional, string) In a PUT New Notification Rule request, use this attribute to specify the service region in which to create the notification rule. To create the rule in the default region set this attribute to the default region name or to empty (""). If you omit the "region" attribute in a PUT, then by default the notification rule is created in the default region.

In a GET Notification Rules response, the "region" attribute will be present but set to empty. The client application will be aware of what region the retrieved rules are from because the desired region is specified in the GET request line.

 ruleId

(Mandatory, string) System-generated unique ID for this rule. For the default notification rules that come packaged with the HyperStore system this will be a simple integer like "1", "2", or "14". For rules that you create yourself the system will generate a ruleId in the form of a UUID string like "8e4cc533-360a-4dd5-bfe4-6b5f5b6c40da".

In a PUT (when you are creating a new rule), include the "ruleId" attribute and set it to empty (""). The system will subsequently generate a ruleId upon rule creation.

In a POST (when you are updating an existing rule), set the "ruleId" attribute to the ruleId of the rule you want to update. To find out what the ruleId is for a particular rule, use the GET Notification Rules method.

 snmpTrap

(Optional, boolean) Whether to transmit an SNMP trap as part of the notification when this rule is triggered, true or false. If a trap is sent it is sent to the destination configured on the CMC page Cluster → Cluster Config → Configuration Settings.

This defaults to false if the attribute is omitted in a PUT New Notification Rule request.

 statId

(Mandatory, string) ID of the node statistic being checked for this rule. See the next table below for a list of supported statIds.

The table below lists statistics for which notification rules can be defined. These statistics are monitored on each node. Note that the sample "conditionVal" column is not intended to suggest suitable values upon which to base notification rules but simply to illustrate the appropriate data format. The right-most column indicates whether a rule for that statId already exists, as part of the default set of notification rules that come pre-packaged with the HyperStore system.

Example

This first example shows a NotificationRule object as it might appear in a PUT request body. It creates a new rule that administrators be notified via email (using the default system administrator email address list) if S3 GET latency on any node exceeds 150 milliseconds. Note that the "ruleId" attribute is included in the request body but it is set to an empty value.

{"email":"default","statId":"s3GetLatency","condition":">","conditionVal":"150","snmpTrap":false,
"ruleId":"","enabled":true,"region":""}

This second example shows a NotificationRule object as it appears in a GET response. It specifies a rule that administrators be notified via email if CPU usage on any node exceeds 90%. This is one of the default notification rules that comes with the HyperStore system.

{"email":"default","statId":"cpu","condition":">","conditionVal":"0.9","snmpTrap":false,
"ruleId":"12","enabled":true,"region":""}
