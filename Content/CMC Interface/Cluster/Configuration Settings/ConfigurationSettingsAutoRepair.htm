Auto-Repair Schedule Settings

[CMC Interface]

The settings in this section of the CMC’s Configuration Settings page configure the scheduling of the HyperStore auto-repair feature. After making any edits, click Save at the bottom of the page to dynamically apply the configuration change to the cluster.

For more information about the auto-repair feature see Automatic Data Repair Feature Overview

Replicas Repair Interval (Minutes)

For each HyperStore node, the interval (in minutes) between scheduled auto-repairs of replicated S3 object data. This feature invokes the hsstool repair operation to automatically repair each node at the configured interval. The nodes will not all be repaired simultaneously, however — instead the start times will be staggered so that only one node is repaired at a time.

Auto-repair of the replicated object data in your cluster is a resource-intensive operation and should be configured to run infrequently.

Default = 129600 (every 90 days)

 EC Repair Interval (Minutes)

For each HyperStore node, the interval (in minutes) between scheduled auto-repairs of erasure coded S3 object data. This feature invokes the hsstool repairec operation to automatically repair each node at the configured interval. The nodes will not all be repaired simultaneously, however — instead the start times will be staggered so that only one node is repaired at a time.

Auto-repair of the erasure coded object data in your cluster is a resource-intensive operation and should be configured to run infrequently.

Default = 128160 (every 89 days)

 Cassandra Repair Interval (Minutes)

For each HyperStore node, the interval (in minutes) between auto-repairs of Cassandra data. This interval is added to each node’s most recent Cassandra repair start time to determine a scheduled next start start time for Cassandra auto-repair on that node.

The Cassandra auto-repair feature must be less than 10 days. It’s recommended that you leave it at its default of seven days.

Default = 10080 (every seven days)
