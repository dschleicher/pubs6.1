Consistency Levels Definitions

When setting custom consistency levels for object data or metadata (through either the CMC's Storage Policies page or the Admin API's PUT New Storage Policy method) you can choose from among the consistency levels described below. Your options for supported consistency levels will be limited by the data distribution type (replication or erasure coding) and by whether you have a single data center or multiple data centers.

The consistency levels descriptions indicate the requirements that are imposed by each consistency level, in terms of what must happen before the system can return a success response to the S3 client. If the requirement cannot be met, an HTTP 503 error response is returned to the client. For example if you configure a storage policy to use consistency level EACH ALL, then for a given read (S3 GET) or write (S3 PUT) request associated with a bucket that's using that policy an HTTP 503 will be returned to the client if the requirements of EACH ALL cannot be met for that request.

In the case of writes, if the requirement is met by something less than completing writes of all replicas (or all fragments in the case of erasure coded data), then after returning a success response to the client the system continues to try to complete the remaining writes. If these fail they will later be recreated by automatic data repair.

For replicated object data, you can optionally configure separate consistency levels for writes of new objects ("Write New" in the configuration interface) as versus overwrites (updates) of existing objects ("Write" in the interface).

For erasure coded object data, there are no configurable settings for reads. Instead, object data reads simply require that "k" unique fragments are available in the system. In a multi-DC context, some of the "k" unique fragments can come from a remote DC if not enough are available in the local DC.

For diagrams illustrating how consistency levels work, see Availability for Replicated Object Writes.

What "QUORUM" Requires Before Success Response Can Be Returned to the Client

 

Example

 

Replicated objects or metadata

 

Reads or writes

 Single The operation must succeed for a majority of replicas in the single DC. If the storage policy uses 3X replication, then the operation must succeed for 2 replicas.
Multiple The operation must succeed for a majority of the combined number replicas in all DCs. If the storage policy uses 3X replication in DC-East and 2X replication in DC-West, then the operation must succeed for 3 replicas. The 3 successes could all be in DC-East; or 2 in DC-East and 1 in DC-West; or 1 in DC-East and 2 in DC-West.

Mathematically, a quorum or majority for these purposes is defined as (#replicas/2) +1, rounded down to the nearest integer. For example, from a set of 5 replicas a quorum is constituted by 3 replicas (5/2 = 2.5 + 1 = 3.5 rounded down = 3); and from a set of 6 replicas, a quorum is constituted by 4 replicas (6/2 = 3 + 1 = 4 rounded down = 4).

Replicated objects

 

Reads or writes

 Single The operation must succeed for only one replica in the single DC. If the storage policy uses 3X replication, then the operation must succeed for 1 replica.
Multiple The operation must succeed for only one replica in either DC (one replica in total). If the storage policy uses 3X replication in DC-East and 2X replication in DC-West, then the operation must succeed for 1 replica in either DC-East or DC-West.

As an advanced option for replicated data or metadata, you could also specifiy multiple consistency levels to create a cascading, "best effort" system behavior in regard to consistency requirements. For information about this option see Dynamic Consistency Levels.
