Availability for Erasure Coded Object Reads

[Diagram]

The read scenarios in this section continue the presumptions of the write scenarios in Erasure Coding -- Write Availability: 4+2 erasure coding is being used to protect objects in a nine node cluster, in a single data center. We are now trying to read the same object that was written in the preceding section: object data fragments are stored on node1, node2, node5, node6, node7, and node8; and object metadata replicas are stored on node2, node3, node4, node5, and node7. We are presuming data consistency, which is the typical condition.

When you create storage policies for erasure coding in a single DC there is only one supported consistency level (CL) requirement for S3 object data read: LOCAL QUORUM. There is no need for a stricter object data read policy. You can separately configure CL requirements for S3 object metadata replica reads, choosing from ALL or QUORUM.
