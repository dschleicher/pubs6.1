Availability for Replicated Object Reads

[Diagram]

The read scenarios in this section continue the presumptions of the write scenarios from Availability for Replicated Object Writes: 3X replication is being used to protect objects in a six node cluster, in a single data center. We are now trying to read the same object that was written in the preceding section: object data replicas are stored on node1, node5, and node6; and object metadata replicas are stored on node2, node3 and node5.

When you configure storage policies for replication in a single DC you can choose one of three supported consistency level (CL) requirements for S3 object data reads: ALL, QUORUM, or ONE. An object data read CL of ONE is sufficient to ensure that only fresh data is returned to clients (when combined with an object metadata read CL of QUORUM or stricter), and so the following scenarios assume an object data read CL of ONE. You can separately configure CL requirements for object metadata reads, choosing from ALL or QUORUM.

Note  The read scenarios that follow presume that all the object data replicas and all the object metadata replicas are consistent (having the same timestamp), which is the usual condition. For information about read availability if data is inconsistent, see Data Freshness for Replicated Object Reads.
