UserInfo

[Admin API Object]

iThe UserInfo object is returned as the response payload for these Admin API operations:

  • GET User
  • GET User List

The UserInfo object is required as the request payload for this Admin API operation:

  • POST Updated User

The UserInfo object is required as the request payload for this Admin API operation, and then is also returned as the response payload:

  • PUT New User

The UserInfo object consists of the attributes below. Each attribute description indicates whether the attribute is mandatory or optional when this object is transmitted as a request payload.

active

(Optional, string) Whether the user is active — "true" or "false".

  • "true" indicates an active user.
  • "false" indicates that the user is not an active user. Non-active users are users who are in one of these statuses:
    • Inactive — These users have had their status set to inactive via the POST Updated User method (with UserInfo object attribute "active" set to false). These users' stored S3 objects still exist, but their S3 access credentials have been deactivated and they cannot log into the CMC.
    • Deleted — These users have been deleted from the S3 service via the DELETE User method. Their S3 access credentials have been deleted, and their S3 buckets and objects have been deleted and are unrecoverable. (These users can’t be retrieved through GET User — they can only be retrieved through GET User List.)
    • Deleting — These users are in the process of being deleted from the S3 service via the DELETE User method. The deletion process for these users has not yet completed. (These users can’t be retrieved through GET User — they can only be retrieved through GET User List.)

If the "active" attribute is unspecified for a PUT New User operation, it defaults to "true". If the "active" attribute is unspecified for a POST Updated User operation, the user will retain their existing status.

address1

(Optional, string) User’s street address line 1

address2

(Optional, string) User’s street address line 2

canonicalUserId

(Optional, string) Canonical user ID, globally unique within the HyperStore system. This is automatically generated by the system when a new user is created.

  • For users created prior to HyperStore 3.0 (before the canonical ID feature existed), the canonicalUserId is the user’s <groupId>|<userId> combination.
  • For users created in HyperStore 3.0 and later, the canonicalUserId is a system-generated, globally unique printable string. The canonicalUserId is unique per user across the system and over time, even in the case where a user with a specific <groupId>|<userId> combination is deleted from the system and then a new user is subsequently added with the same <groupId>|<userId> combination. The new user will be assigned a different canonicalUserId than the deleted user. This allows past and present users to be uniquely identified for purposes such as usage reporting.
  • Client must not supply a canonicalUserId in a PUT New User request and does not need to supply one in a POST Updated User request.
city

(Optional, string) User’s city

country

(Optional, string) xUser’s country

emailAddr

(Optional, string) User’s email address

fullName

(Optional, string) User’s full name. Maximum length 64 characters.

groupId

(Mandatory, string) Group ID of the group to which the user belongs.

Note that the groupId of the system admin group is "0".

phone

(Optional, string) User’s phone number

state

(Optional, string) User’s state

userId

(Mandatory, string) User ID.

  • Only letters, numbers, dashes, and underscores are allowed.
  • Length must be at least 1 character and no more than 64 characters.
  • The following IDs are reserved for system use and are not available to individual users: "anonymous", "public", "null", "none", "admin", "0".
userType

(Mandatory, string) User type. One of {"User","GroupAdmin","SystemAdmin"}.

website

(Optional, string) User’s website URL.

zip

(Optional, string) User’s zip code
