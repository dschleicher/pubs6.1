Dashboard

[CMC Interface]

Path: Dashboard

Supported task:

  • View summary service status and drill down for details

The CMC’s Dashboard provides a high-level view of the status of your HyperStore object storage service. If you have multiple service regions, there is a separate dashboard view for each region.

Along with conveying summary information, each graphical element of the dashboard is hyper-linked to a relevant section of the CMC interface for more detail.

Description

Across the whole service region, the percentages of total data disk capacity that are currently Reserved, Used, or Free. To see the percentage numbers slide your cursor over each portion of the tri-colored circle.

Note  The Reserved space is the disk space that’s set as "reserved-blocks-percentage" in your Linux system configuration (the portion of the disk space that’s reserved for privileged processes). By default in Linux the configurable "reserved-blocks-percentage" for a file system is 5% of disk capacity. In a HyperStore Appliance it’s set to 0%.

The displayed "Total" figure is the total disk capacity that is dedicated to HyperStore data storage throughout the service region. Only disks that store HyperStore data directories (as configured by the hyperstore_data_directory setting in common.csv) or the Cassandra data directory (as configured by the cassandra_data_directory setting in common.csv) count toward these statistics. These statistics do not count disks that are being used only for OS or application files, for example.

Below the "Total" figure is a graph that shows the moving proportion of free to used capacity over the past 30 days.

Links to the Capacity Explorer page, where you can see a breakdown of your available capacity by region, data center, and node.
Description

This section displays a green circle labeled "Healthy" if no services are currently down in the cluster. (Note that all the nodes in a service region constitute one "cluster", even if there are multiple data centers within the region).

An orange circle labeled "Check Services" displays if any of the Hyperstore services are down on any node within the region.

This section also indicates the current number of unacknowledged alerts (triggered by configurable alert rules) in the region, if any.

Links to the Alerts page, where you can review and acknowledge node alerts.
Description

This graph shows the number of S3 transactions processed per second, across the whole service region, over the past 30 days. This data is broken out to PUT transactions (in blue) and GET transactions (in green). To highlight just the PUT trend line or just the GET trendline, hold your cursor over the "PUT" or "GET" label to the upper right of the graph. Or to display the numerical value corresponding to a particular point in time, hold your cursor over that point in time on the trend line.

These S3 stats are recalculated each five minutes, based on the most recent five minutes of transaction data. HEAD transactions are counted toward the GET stat, and POST transactions are counted toward the PUT stat.

Links to the Cluster Usage page. This page shows the same graph but in somewhat larger form.
Description

This graph shows data throughput for S3 transactions across the whole service region, over the past 30 days. This data is expressed as MB per second and is broken out to PUT transactions (in blue) and GET transactions (in green). To highlight just the PUT trend line or just the GET trendline, hold your cursor over the "PUT" or "GET" label to the upper right of the graph. Or to display the numerical value corresponding to a particular point in time, hold your cursor over that point in time on the trend line.

These S3 stats are recalculated each five minutes, based on the most recent five minutes of transaction data. HEAD transactions are counted toward the GET stat, and POST transactions are counted toward the PUT stat.

Links to the Cluster Usage page. This page shows the same graph but in somewhat larger form.
Description

Number of users in your whole HyperStore system. If your system encompasses multiple service regions, users are registered to the system as a whole (they are not tied to a particular region).

Along with regular S3 service users, the total number includes system admin users and group admin users.

Links to the Manage Userss page, where you can create new users or retrieve and edit existing users.
Description

Number of user groups in your whole HyperStore system. If your system encompasses multiple service regions, user groups are registered to the system as a whole (they are not tied to a particular region).

Links to the Manage Groups page, where you can create new groups or retrieve and edit existing groups.
Description

Number of S3 objects stored in this service region. If you have multiple service regions, each region has an entirely different and independent inventory of objects.

S3 objects that have been auto-tiered to Amazon S3, Amazon Glacier, or other HyperStore regions or systems do count toward this stat.

In the case of versioned objects, each version of the object counts towards this stat.

This statistic counts each object, not each replica — for example, if an object is replicated three times across your cluster (to protect data durability and availability), this counts as one object, not three.

Links to the Object Locator page, where you can retrieve information about where a particular object is stored within your cluster.
Description

Number of nodes in this service region.

Links to the Node Status page, where you can see status details for individual nodes, start and stop HyperStore services on nodes, and review and acknowledge node alerts.
Description

Number of data centers in this service region. Each service region can encompass multiple data centers, with data replication across data centers.

Links to the Data Centers page, where you can see your HyperStore node inventory in each of your data centers.
Description

HyperStore software version used in this service region.

Links to the Cluster Information page, where you can view static information about your cluster such as license information, the S3 service endpoint for your system, and the mapping of specialized service roles (such as the Redis QoS database master role) to individual nodes.
