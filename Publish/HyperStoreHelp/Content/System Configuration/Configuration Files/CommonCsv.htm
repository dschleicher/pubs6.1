common.csv

[Configuration File]

Note  The HyperStore system’s interactive installation script cloudianInstall.sh writes to this file. All common.csv settings that require environment-specific customization are automatically pre-configured by the install script, based on information that you provided during the installation process. It is not mandatory to make manual edits to this file, but the file does include settings that you may wish to fine tune for your particular circumstances and preferences.

The common.csv file is the main Puppet extdata file. It configures basic settings for all the major HyperStore components. On the Puppet master the path to the file is:

/etc/cloudian-<version>-puppet/manifests/extdata/common.csv

The settings in common.csv are grouped according to which HyperStore component they configure. The documentation that follows describes the settings in each section of common.csv, in the order in which they appear in the file.

GENERAL Section
release_version

Default = 6.0

Do not edit.

cloudian_license_file_name

Name of your Cloudian license file.

Default = Set during installation based on operator input.

Do not edit manually. To apply an updated license file, use the CMC’s Update License function (Cluster → Cluster Config → Cluster Information → Update License). That function will automatically update this configuration setting as appropriate and dynamically apply the change to your live system. No service restart is necessary.

default_region

The default service region for your S3 service. Must be one of the regions listed for the regions setting. In a multi-region HyperStore system, the default region plays several roles in the context of S3 storage bucket LocationConstraint functionality. For example:

  • For PUT Bucket requests that lack a CreateBucketConfiguration element in the request body, the bucket will be created in the default region.
  • For PUT Bucket requests that do include a CreateBucketConfiguration element (with LocationConstraint attribute), the PUT requests typically resolve to the default region, and S3 Service nodes in the default region then initiate the process of creating the bucket in the requested region.

If your HyperStore system has only one service region, make that region the default region.

Default = Set during installation based on operator input.

Do not change this setting after your system is installed and running. If for some reason you need to change which region is your default region, please consult with Cloudian Support.

regions

Comma-separated list of service regions [XREF] within your HyperStore system. Region names must be lower case with no dots, dashes, underscores, or spaces. Even if you have only one region, you must give it a name and specify it here.

Default = Set during installation based on operator input.

HyperStore SERVICE Section
hyperstore_data_directory

A quote-enclosed, comma-separated list of mount points to use for S3 object storage.

In a production environment, use dedicated disks for S3 object storage. Do not use the same disks that are storing the cassandra_commit_log_directory [XREF] and cassandra_data_directory [XREF]. For more information about HyperStore disk and mount point requirements, see File System [XREF].

The system will automatically assign virtual nodes (vNodes) to each of your S3 data mount points, in a manner that allocates an approximately equal total token range to each mount point. For more information about HyperStore vNodes see HyperStore vNodes [XREF]. Note that the HyperStore system allocates an approximately equal total token range to each S3 data mount point even if the mount points are associated with disks of different sizes.

Do not change the hyperstore_data_directory setting once your cluster is operational.

Do not use symbolic links when specifying your mount points for the hyperstore_data_directory setting. The HyperStore system does not support symbolic links for these directories.

Example of a multiple mount point configuration = "/cloudian1,/cloudian2,/cloudian3,/cloudian4"

Default = Set during installation based on operator input, if host has multiple disks. For hosts with only one disk, default is /var/lib/cloudian

cloudian_s3admin_min_threads

Minimum number of threads to keep in each Admin Service node’s HTTP request processing thread pool.

The Admin Service uses a thread pool to process incoming HTTP requests from clients. An initial pool of threads is created at server initialization time, and additional threads may be added if needed to process queued jobs. Idle threads are terminated if not used within a thread timeout period — unless the number of threads in the pool is down to cloudian_s3admin_min_threads. If only this many threads are in the pool, then threads are kept open even if they’ve been idle for longer than the thread timeout.

Default = 10

To apply change, after Puppet propagation restart S3 Service

cloudian_s3admin_max_threads

Maximum number of threads to allow in the Admin Service’s HTTP request processing thread pool. If there are fewer than this many threads in the pool, new threads may be created as needed in order to handle queued HTTP request processing jobs. If the maximum thread pool size is reached, no more threads will be created — instead, queued HTTP request processing jobs must wait for an existing thread to become free.

Default = 50

cassandra_num_tokens

The number of storage ring tokens to assign to each host machine. Each token on the host machine constitutes a "virtual node" (vNode) within the storage token ring that encompasses all of the hosts on which you are running Cassandra and the HyperStore Service. This token ring is used to determine the hosts on which to store and replicate each S3 object, based on a consistent hashing scheme. The Cassandra, HSFS replication, and erasure coding storage layers all utilize the same token ring. For more information on HyperStore vNodes, see HyperStore vNodes [XREF].

Maximum allowed value is 256.

During installation (using the cloudianInstall.sh script), the value of this configuration property is automatically set to 8 X the number of HyperStore data mount points [XREF] on the host, but no less than 32 and no more than 256. For example:

  • 1 mount point → 32 tokens assigned to host
  • 2 mount points → 32 tokens assigned to host
  • 3 mount points → 32 tokens assigned to host
  • 4 mount points → 32 tokens assigned to host
  • 5 mount points → 40 tokens assigned to host
  • 6 mount points → 48 tokens assigned to host
  • 10 mount points → 80 tokens assigned to host
  • 32 mount points → 256 tokens assigned to host
  • 33 mount points → 256 tokens assigned to host
  • 40 mount points → 256 tokens assigned to host

The virtual nodes assigned to a host will be allocated across the host’s HyperStore mount points in a manner that gives each mount point an approximately equal total token range. So, while the total number of vNodes allocated to a host will be 8 X the number of mount points (within the bounds stated above), there will not necessarily be 8 vNodes assigned to each mount point. For example some mount points may have 7 vNodes while others have 9, depending on the token range sizes associated with the individual vNodes. What will be approximately equal is the aggregate token range size allocated to each mount point.

Note  Each individual vNode is associated with a token range of a particular size — that is, the numerical range (the delta) between that vNode’s token number and the next-lower token number from among all the tokens in the cluster. The variance in the token range size associated with individual vNodes comes from the fact that Cassandra randomly generates tokens, within an integer token space ranging from 0 to 2127-1.

Default = Set during installation based on operator input regarding the number of mount points.

IMPORTANT: Do not edit this setting manually. Do not change it before the installation or after your cluster is in operation.

cmc_cloudian_admin_user

For the CMC, the login user name of the default system admin user. The system admin will use this user name to log into the CMC.

Default = admin

This cannot be changed through configuration.

Note  The default password for the default CMC system admin user is "public". You can change this password when you are logged into the CMC as the default admin user. Hold your cursor over the user name that displays in the upper right of the screen (by default, "Admin"), then select the Security Credentials option and change the password.

cmc_domain

Fully qualified domain name for the CMC. This FQDN must be resolvable for CMC clients.

Default = Set during installation based on operator input.

To change this, use the HyperStore installer’s Advanced Installation Configuration Options [XREF].

cmc_web_secure

Whether the CMC should require HTTPS for all client connections, true or false.

Set this property to "true" to require HTTPS. In this mode of operation, requests incoming to the CMC’s regular HTTP port will be redirected to the CMC’s HTTPS port.

Set this property to "false" to allow clients to connect through regular HTTP. In this mode of operation, requests incoming to the CMC’s HTTPS port will be redirected to the CMC’s regular HTTP port.

Default = true

To apply change, after Puppet propagation restart CMC.

cmc_http_port

Port on which the CMC listens for regular HTTP requests.

Default = 8888

To apply change, after Puppet propagation restart CMC.

cmc_https_port

Port on which the CMC listens for HTTPS requests.

Default = 8443

To apply change, after Puppet propagation restart CMC.

cmc_admin_secure_port

Port on which the Admin Service listens for HTTPS requests. The CMC submits subscriber payment information requests to this secure port, if you use the payment information feature.

Default = 19443

To apply change, after Puppet propagation restart CMC.

cmc_admin_secure_ssl

Set to "true" to require that the Admin Service’s certificate be CA validated.

Set to "false" to allow the Admin Service’s certificate to be self-signed.

Default = false

To apply change, after Puppet propagation restart CMC.

cmc_application_name

Name of the CMC web application, to be displayed in the URL path for the application. By default, the CMC’s URL paths are in the form

https://<host>:<port>/Cloudian/<page>.htm.

Default = Cloudian

IMPORTANT: In the current release the CMC application name cannot be customized. It must be left at "Cloudian".
