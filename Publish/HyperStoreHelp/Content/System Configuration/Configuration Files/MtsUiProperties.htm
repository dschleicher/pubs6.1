mts-ui.properties.erb

[Configuration File]

The mts-ui.properties file is the primary configuration file for the Cloudian Management Console server (CMC). On your CMC nodes, the file is located at the following path by default:

/opt/tomcat/webapps/Cloudian/WEB-INF/classes/mts-ui.properties

On the Puppet master node, the template file for mts-ui.properties is mts-ui.properties.erb:

/etc/cloudian-<version>-puppet/modules/cmc/templates/mts-ui.properties.erb

Certain mts-ui.properties.erb properties take their values from settings in Puppet extdata files (most often common.csv) or from settings that you can control through the CMC's Configuration Settings page. In the mts.properties.erb file these properties' values are formatted as bracket-enclosed variables, like <%= … %>. In the property documentation below, the descriptions of such properties indicate "Takes its value from <location>: <setting>; use that setting instead.". For example, "Takes its value from common.csv: hyperstore_data_directory; use that setting instead."). Also in such cases, a link to the source setting's description is provided.

IMPORTANT: After making changes to mts-ui.properties.erb use the HyperStore installer (cloudianInstall.sh) on the Puppet master to propagate your changes to the cluster and to restart the CMC Service.

storageuri.ssl.enabled

If this is set to "true", the CMC uses HTTPS to connect to the HyperStore S3 Service (in implementing the CMC Buckets & Objects functionality). If "false", the CMC uses regular HTTP to connect to the HyperStore S3 Service.

Do not set this to "true" unless you have set up SSL for your HyperStore S3 Service [XREF]. By default the HyperStore S3 Service does not use SSL/HTTPS.

Default = false

If you used a CA-verified certificate to implement SSL for your HyperStore S3 Service, then all that is necessary for the CMC to be able to connect to the S3 Service is for you to set mts-ui.properties.erb: storageuri.ssl.enabled to "true" on the Puppet master node, have Puppet push the change out to the cluster, and restart your CMC service.

However, if you used a self-signed certificate to implement SSL for your HyperStore S3 Service then additional action is necessary because the AWS-SDK that the CMC uses to implement its S3 client does not by default support connecting to an S3 service that is using a self-signed SSL certificate. The full procedure is:

  1. Set mts-ui.properties.erb: storageuri.ssl.enabled to "true" on the Puppet master node.
  2. Have Puppet push the change out to the cluster.
  3. On each individual CMC node use the following command to create a file /opt/tomcat/bin/setenv.sh with a single configuration line in it which disables certificate checking by the CMC’s S3 client:
echo "CATALINA_OPTS=-Dcom.amazonaws.sdk.disableCertChecking=true" > /opt/tomcat/bin/setenv.sh

You cannot use Puppet to disseminate this configuration change; you must do it manually on each CMC node.

  1. Restart the CMC service.
path.style.access

Whether the CMC should use "path style" request formatting when submitting S3 requests to the HyperStore S3 Service. In path style S3 requests, the bucket name is part of the URI rather than being part of the Host header value.

Options are:

  • true — The CMC will use path style HTTP request formatting when submitting S3 requests to the HyperStore S3 Service. The bucketname associated with the request will be in the request URI. For example:
PUT /bucket1/objectname HTTP/1.1
Host: s3-region1.mycompany.com
  • false — The CMC will not use path style HTTP request formatting when submitting S3 requests to the HyperStore S3 Service. The bucketname associated with the request will be in the Host header. For example:
PUT /objectname  HTTP/1.1
Host: bucket1.s3-region1.mycompany.com

With this approach (which is what Amazon S3 currently uses) your DNS environment must be configured to resolve this type of Host value. See DNS Set-Up.

Default = false

s3.upload.part.minsize

When the CMC uses Multipart Upload to transmit an object to the S3 Service, each of the parts will be this many bytes or larger — with the exception of the final part, which may be smaller. For example, if Multipart Upload is used for an 18MB object, and the configured minimum part size is 5MB, the object will be transmitted in four parts of size 5MB, 5MB, 5MB, and 3MB.

Default = 5242880 (5MB)

s3.upload.part.threshold

When a CMC user uploads an object larger than this many bytes, the CMC uses Multipart Upload to transmit the object to the S3 Service, rather than PUT Object.

Default = 16777216 (16MB)

admin.manage_users.enabled

This setting controls whether the Manage Users function will be enabled in the CMC GUI. For a screen shot and description of this function, see Manage Users[XREF] in the CMC Help document.

Options are:

  • true — This function will display for users logged in as a system administrator or group administrator. For group admins this function is restricted to their own group.
  • false — This function will not display for any users. If you set this to "false" then the Manage Users functionality as a whole is disabled and the more granular admin.manage_users.*.enabled properties below are ignored.
  • SystemAdmin — This function will display only for users logged in as a system administrator.
  • GroupAdmin — This function will display only for users logged in as a group administrator.

Default = Commented out and uses internal default of "true". To assign a different value, add an uncommented instance of the setting to mts-ui.properties.erb.

Note  If you want to enable some aspects of the Manage Users function and not others, you can have admin.manage_users.enabled set so that the function is enabled for your desired user types, and then use the granular admin.manage_users.*.enabled properties below to enable/disable specific capabilities.

IMPORTANT: If you enable CMC SSO functionality (using the sso.enabled[XREF] setting), then for security reasons you should set a custom value for sso.cookie.cipher.key. Do not leave it at its default value.

Note  If you change this value after CMC SSO has already been in service, end users who had used SSO previously will have on their browser a Cloudian SSO cookie that is no longer valid. If such users access the CMC after your sso.cookie.cipher.key change, the CMC detects the invalid cookie, deletes it, and drops a new, valid one.

Pass phrase to use when the CMC connects via SSH to the Puppet master node or HyperStore nodes when implementing node management functions. The Cloudian installation script automatically populates this setting.

local.temp.dir

This setting supports the feature whereby an updated HyperStore license file can be uploaded via the CMC. Do not edit.

remote.ssh.user

The user as which to connect via SSH to the Puppet master node or HyperStore nodes.

Default = root

remote.ssh.port

The port to which to connect via SSH to the Puppet master node or HyperStore nodes.

Default = 22

uninstall.node.options

Options used by the "Uninstall Node" function on the CMC’s Node Management page. These options specify that HyperStore packages are to be removed from the target node as part of the uninstall process. This setting is for internal system use and should not be edited unless you first consult with Cloudian Support.

Default = -u -r
