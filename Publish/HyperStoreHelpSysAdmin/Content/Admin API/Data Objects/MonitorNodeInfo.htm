MonitorNodeInfo

[Admin API Object]

The MonitorNodeInfo object is returned as the response payload for this Admin API operation:

  • GET Monitoring Data for Node

The MonitorNodeInfo object consists of the attributes below.

Note  Within the MonitorNodeInfo object, all statistics of type MonitorStat are formatted as
"<statName>":{"timestamp":"<UTCMilliseconds>","value":"<statValue>"}.

adminHeapMax

(MonitorStat) The maximum JVM heap size allocated to the Admin Service.

adminHeapUsed

(MonitorStat) The Admin Service’s current JVM heap memory usage on the node.

cassCMSGCCount

(MonitorStat) The number of concurrent mark-sweep (CMS) garbage collections executed since the last start-up of the Cassandra service on this node. This collection type targets old-generation objects.

cassCMSGCTime

(MonitorStat) The aggregate time (in milliseconds) spent on executing CMC garbage collections since the last start-up of the Cassandra service on this node.

cassCopyGCCount

(MonitorStat) The number of Copy garbage collections executed since the last start-up of the Cassandra service on this node.

cassCopyGCTime

(MonitorStat) The aggregate time (in milliseconds) spent on executing Copy garbage collections since the last start-up of the Cassandra service on this node.

cassHeapMax

(MonitorStat) The maximum JVM heap size allocated to the Cassandra Service.

cassHeapUsed

(MonitorStat) The Cassandra Service’s current JVM heap memory usage on the node.

cassParNewGCCount

(MonitorStat) The number of parallel new-generation (ParNew) garbage collections executed since the last start-up of the Cassandra service on this node.

cassParNewGCTime

(MonitorStat) The aggregate time (in milliseconds) spent on executing ParNew garbage collections since the last start-up of the Cassandra service on this node.

cpu

(MonitorStat) Current CPU utilization on the node.

diskAvailKb

(MonitorStat) On the node, the total mounted disk space remaining still available for Cassandra data directory or HyperStore data directory storage. A disk’s "reserved-blocks-percentage" (the portion of the disk that’s reserved for privileged processes) does not count as available space.

disksInfo

(DiskMonitorStat) Current information about the disks on the node. Formatted as {"timestamp":"<UTCMilliseconds>","disks":List<DiskInfo>}.

diskIORead

(MonitorStat) Across all the node’s disks that are being used for Cassandra and HyperStore data directories, the average disk read throughput in bytes per second. This stat is recalculated each minute, based on the most recent minute of data.

diskIOWrite

(MonitorStat) Across all the node’s disks that are being used for Cassandra and HyperStore data directories, the average disk write throughput in bytes per second. This stat is recalculated each minute, based on the most recent minute of data.

diskTotalKb

(MonitorStat) On the node, the total size of the disks mounted for Cassandra data directories or HyperStore data directories (the HyperStore File System [HSFS] and erasure coding file system). This stat does not include other any other disks on your HyperStore hosts — such as those that are being used only for the OS or application files, for example.

ioRx

(MonitorStat) The aggregate network bytes per second received by the node, for all types of network traffic including but not limited to S3 request traffic. This stat is recalculated each minute, based on the most recent minute of data.

ioTx

(MonitorStat)  The aggregate network bytes per second transmitted by the node, for all types of network traffic including but not limited to S3 request traffic. This stat is recalculated each minute, based on the most recent minute of data.

hyperStoreHeapMax

(MonitorStat) The maximum JVM heap size allocated to the HyperStore Service.

hyperStoreHeapUsed

(MonitorStat) The HyperStore Service’s current JVM heap memory usage on the node.

s3GetLatency

(MonitorStat) On the node, the average latency for completing S3 GET transactions, in milliseconds. This stat is recalculated each five minutes, based on the most recent five minutes of transaction data. HEAD transactions are counted toward this stat also.

s3GetThruput

(MonitorStat) On the node, the data throughput for S3 GET transactions, expressed as MB per second. This stat is recalculated each five minutes, based on the most recent five minutes of transaction data. HEAD transactions are counted toward this stat also.

s3GetTPS

(MonitorStat) On the node, the number of S3 GET transactions processed per second. This stat is recalculated each five minutes, based on the most recent five minutes of transaction data. HEAD transactions are counted toward this stat also.

s3HeapMax

(MonitorStat) The maximum JVM heap size allocated to the S3 Service.

s3HeapUsed

(MonitorStat) The S3 Service’s current JVM heap memory usage on the node.

s3PutLatency

(MonitorStat) On the node, the average latency for completing S3 PUT transactions, in milliseconds. This stat is recalculated each five minutes, based on the most recent five minutes of transaction data. POST transactions are counted toward this stat also.

s3PutThruput

(MonitorStat) On the node, the data throughput for S3 PUT transactions, expressed as MB per second. This stat is recalculated each five minutes, based on the most recent five minutes of transaction data. POST transactions are counted toward this stat also.

s3PutTPS

(MonitorStat) On the node, the number of S3 PUT transactions processed per second. This stat is recalculated each five minutes, based on the most recent five minutes of transaction data. POST transactions are counted toward this stat also.

status

(ServiceStatus) xStatus of the node. Formatted as {"ipaddr":"<empty>", "status":[<list of one or more of "OK","SVC_DOWN","LOG_WARN", or "LOG_ERR">],"value":"<sameAsStatus>", "timestamp":"<UTCMilliseconds>"}.

The "ipaddr" value will be empty or null here; an IP address is specified only in the service-specific status attributes (such as "svcCassandra") described toward the bottom of this table.

Possible values within the "status" list are:

  • "OK" — All HyperStore services are up and running on the node, and the node has no unacknowledged events.
  • "SVC_DOWN" — One or more HyperStore services (Admin, Cassandra, HyperStore, Redis QoS, Redis Credentials, Redis Monitor, or S3) is down on the node
  • "LOG_WARN" — There are unacknowledged warnings in an application log on the node (such as the S3 Service application log or the Cassandra application log).
  • "LOG_ERR" — There are unacknowledged errors in an application service log.

The "value" attribute will be identical to the "status" attribute, except formatted as a single string rather than a list of strings. For example:

"status":["SVC_DOWN","LOG_WARN"],"value":"[SVC_DOWN, LOG_WARN]"
svcAdmin

(ServiceStatus) Admin service status. Formatted as {"ipaddr":"<nodeIPAdress>","status":[<list of one or more of "OK","SVC_DOWN", "LOG_WARN", or "LOG_ERR">],"value":"<statusValueFormattedAsString>","timestamp":"<UTCMilliseconds>"}. The other service status attributes that follow below (svcCassandra and so on) are formatted in the same way.

svcCassandra

(ServiceStatus) Cassandra service status.

svcHyperstore

(ServiceStatus) HyperStore service status.

svcRedisCred

(ServiceStatus) Redis Credentials service status.

svcRedisMon

(ServiceStatus) Redis Monitor service status.

svcRedisQos

(ServiceStatus) Redis QoS service status.

svcS3

(ServiceStatus) S3 service status.
